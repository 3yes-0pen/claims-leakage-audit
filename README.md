# Claims Leakage Audit Engine

> Every denied dollar has a birth certificate. This project traces it.

A healthcare revenue cycle analytics project built on synthetic claims data — demonstrating
how denial pattern analysis, failure node attribution, and intervention modeling can surface
the clinical and operational signals hidden inside standard RCM data.

**The Purpose of this Project**
Practice and Exercise on the nuances of insight within the entire Claims, Denials, Remittance cycle. 

Some questions I had that inspired the structure:
- From different perspectives of this same data, what is valuable to see?
- What pieces of information are actually meaningful in this domain?
- What structures and systems conceptualize the solutions and improve the live diagnostics?

## The analytical thesis

A standard RCM dashboard answers only "how much was denied." This project answers five more questions:

| Panel | Audience | The Question |
|---|---|---|
| Revenue Leakage Waterfall | Executive, CFO | How much leaked? |
| Failure Node × Payer/Provider Heatmap | RCM, Contracting | Who caused it? |
| Urgency Triage Queue | Billing Ops, RCM | What do I work today? |
| Denial Pattern Fingerprint by Specialty | Clinical Ops, ACO | What does it mean clinically? |
| Intervention Prioritization Matrix | QI, Clinical Ops, Executive | What do I fix first? |

The key insight: CO-57 concentration in Oncology isn't a billing team failure — it's a
care coordination gap where imaging orders don't trigger auth workflows at the point of
order entry. The intervention belongs upstream of the billing team entirely. This project
is built to surface those distinctions.

## Data layer

The dashboards read from pre-exported parquet files in `data/`. These are generated by
the final cells of each notebook — the notebooks do all the analytical heavy lifting;
the dashboards are purely display and filtering logic.

The raw database (`data/claims.duckdb`, ~689MB) and raw Synthea data are excluded from
the repo (see `.gitignore`). The parquet files are the committed data layer — sufficient
to run both dashboards without any local database.

See `data/MANIFEST.md` for the grain, source notebook, and row count of every file.

## Running locally

```bash
# Clone and set up
git clone https://github.com/3yes-0pen/claims-leakage-audit.git
cd claims-leakage-audit
python -m venv venv && source venv/bin/activate
pip install -r requirements.txt

# Run the 5-panel dashboard
streamlit run dashboard_5panel/landing_page.py

# Run the 3-panel executive dashboard
streamlit run dashboard_3panel/landing_page.py
```

Both dashboards run immediately from the committed parquet files — no database required.

## Notebooks

The notebooks document the full analytical pipeline. Each is self-contained and runs
against `data/claims.duckdb` (not included in the repo — regenerate from `src/` or
contact me for the dataset).

| Notebook | What it does |
|---|---|
| `03_denial_forensics` | Builds `denial_forensics_v2` — priority scores, failure node attribution, triage queue |
| `04_ar_workflow_view` | AR aging analysis, charge lag patterns, adjudication cycle time |
| `05_intervention_modeling` | Simulates dollar impact of each intervention against target denial rates |
| `06_pop_health_view` | CARC composition by specialty — denial pattern fingerprint |
| `07_financial_integrity_view` | Revenue waterfall, NCR by payer, contractual vs. denial leakage |

- **DuckDB** — in-process SQL analytics over synthetic claims data
- **pandas** — DataFrame manipulation and parquet I/O
- **Plotly** — interactive charts (waterfall, heatmap, stacked bar, scatter)
- **Streamlit** — multi-page dashboard framework
- **pyarrow** — parquet serialization layer

---

## About the data

All data is synthetic — generated via Synthea and processed through a custom claims
simulation pipeline (`src/`) that models payer behavior, denial patterns, prior auth
rules, and appeal outcomes across five provider personas and four payer types.

No real patient data was used in this project.
